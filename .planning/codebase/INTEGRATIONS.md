# External Integrations

**Analysis Date:** 2025-01-31

## APIs & External Services

**AWS — Primary Cloud Integration:**
- **DynamoDB** — Task state caching and pipeline execution tracking (cloud mode)
  - SDK/Client: `boto3` (optional extra `kptn[aws]`)
  - Client files: `kptn/caching/client/DbClientDDB.py`, `kptn/caching/client/dynamodb/` (full CRUD operations)
  - Auth: `AWS_REGION` env var + boto3 credential chain (profile, env, instance role)
  - Table name: `DYNAMODB_TABLE_NAME` env var (default: `"tasks"`)
  - Local emulation: `LOCAL_DYNAMODB=true` env var routes to `http://localhost:8000` (DynamoDB Local via `docker-compose-ddb.yml`)

- **ECR (Elastic Container Registry)** — Docker image push/pull for pipeline containers
  - SDK/Client: `boto3` (`kptn/deploy/push.py`, `kptn/deploy/ecr_image.py`)
  - Auth: Temporary AWS credentials fetched from authproxy endpoint; ECR authorization token obtained via `boto3.client("ecr").get_authorization_token()`
  - Repo discovery: `X-Aws-Flow-Runner-Repo` response header from authproxy

- **ECS / AWS Batch** — Pipeline task execution (infrastructure mode via Step Functions)
  - Code generation: `kptn/codegen/lib/stepfunctions.py`
  - Resource ARNs: `arn:aws:states:::ecs:runTask.sync` (default), `arn:aws:states:::batch:submitJob.sync`
  - Env injection: `DYNAMODB_TABLE_NAME` injected into container environment by `kptn/cli/run_aws.py`

- **AWS Step Functions** — Orchestrates multi-task pipeline execution in cloud mode
  - State machine JSON generated by `kptn/codegen/lib/stepfunctions.py`
  - Scaffold templates: `kptn/codegen/templates/stepfunctions/`

- **Authproxy** (internal/self-hosted) — Reverse proxy that issues short-lived AWS credentials
  - Used by: `kptn/deploy/ecr_image.py`, `kptn/deploy/push.py`, `kptn/aws/` module
  - Endpoint resolution: `kptn/deploy/authproxy_endpoint.py`
  - Returns: `AccessKeyId`, `SecretAccessKey`, `Token` in JSON body; `X-AWS-Region`, `X-AWS-Externals-Bucket`, `X-AWS-Artifacts-Bucket` in response headers

**Prefect — Workflow Orchestration (Optional):**
- Used for cloud-deployed pipeline flows
  - SDK/Client: `prefect` 3.4.25 (optional extra `kptn[prefect]`)
  - Integration files: `kptn/deploy/prefect_deploy.py`, `kptn/caching/prefect.py`
  - Auth: `PREFECT_API_URL` env var or Prefect profile settings
  - Work pool: `docker-pool` (default); configurable per deployment
  - AWS credentials threaded through Prefect job variables (ECS/Docker env injection)

## Data Storage

**Databases:**

- **SQLite** — Default local state store for task hashes/execution history
  - Client: `kptn/state_store/sqlite.py` (`SqliteBackend`)
  - Path: `.kptn/kptn.db` (relative to project root; configurable via `kptn.yaml` `settings.db_path`)
  - No external connection required; file-based

- **DuckDB** — Optional analytics state store and primary data format for pipeline outputs
  - Client: `kptn/state_store/duckdb.py` (`DuckDbBackend`), `kptn/caching/client/DbClientDuckDB.py`
  - SDK: `duckdb` 1.4.1 (optional extra `kptn[duckdb]`)
  - Table preview and lineage served by `kptn_server/service.py`
  - Connection resolved from pipeline's runtime config object by `kptn_server/service.py` (`_duckdb_connection_from_runtime_config()`)
  - URI scheme: `duckdb://<path>#<table>` used in task output declarations

- **DynamoDB** — Cloud task state store (see AWS section above)
  - Local emulation: `amazon/dynamodb-local:latest` container, port 8000, data volume at `./dynamodb/`
  - Schema: single-table design with composite PK pattern `BRANCH#{key}#PIPELINE#{id}#TASK#{id}`

**File Storage:**
- **S3** — External artifact storage in cloud mode
  - Referenced via `EXTERNAL_STORE` env var (e.g., `s3://nibrs-estimation-externals`)
  - `ARTIFACT_STORE` env var for local artifact path in containerized runs

**Caching:**
- SQLite and DuckDB double as both state stores and caching backends
- Hash-based change detection: file hashes, SQLite table hashes, DuckDB table hashes computed in `kptn/change_detector/`
- Legacy DynamoDB-backed cache also supported (`kptn/caching/TaskStateCache.py`)

## Authentication & Identity

**Auth Provider:**
- No centralized auth provider (no OAuth, JWT, or session management in application code)
- AWS authentication via standard boto3 credential chain: env vars → AWS profiles → EC2 instance role
- Authproxy pattern: custom internal proxy service issues temporary STS-style credentials to pipeline workers

**Implementation:**
- `kptn/aws/` module: `get_aws_creds_from_prefect_url()` fetches credentials from authproxy at Prefect API host:8080
- AWS profile support: `AWS_PROFILE`, `AWS_SHARED_CREDENTIALS_FILE` env vars in containerized deployments

## VS Code Extension Integration

**Communication Protocol:**
- JSON-RPC 2.0 over stdin/stdout (`kptn_server/api_jsonrpc.py`)
- Extension spawns Python backend process; communicates via newline-delimited JSON messages
- Methods: `getMessage`, `generateLineageHtml`, `getTablePreview`, `getTableColumns`, `getLineageServer`
- Embedded HTTP server started by `start_lineage_server()` for webview content serving (random port on 127.0.0.1)

## Environment Configuration

**Key env vars referenced in code:**
- `DYNAMODB_TABLE_NAME` — DynamoDB table for task state (default: `"tasks"`)
- `AWS_REGION` — AWS region for DynamoDB/ECR/ECS
- `LOCAL_DYNAMODB` — Set to `"true"` to use local DynamoDB emulator
- `PREFECT_API_URL` — Prefect server API base URL
- `EXTERNAL_STORE` — S3 URI for external artifact storage
- `ARTIFACT_STORE` — Local path for artifacts in containerized runs
- `AWS_PROFILE` / `AWS_SHARED_CREDENTIALS_FILE` — AWS credential configuration in Docker
- `PGHOST`, `PGPORT`, `PGPASSWORD`, `PGUSER`, `PGDATABASE` — PostgreSQL connection (referenced in Prefect job variables; not used by core kptn library directly)
- `CYPRESS_BASE_URL` — Override base URL for Cypress E2E tests (default: `http://localhost:8000`)

- `.env` file existence not detected in repo root
